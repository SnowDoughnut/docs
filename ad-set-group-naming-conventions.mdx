<Accordion title="Concept review: ad set / ad group / variant identifiers">
  Date tells you when a customer profile was built or last refreshed, which matters because customer profile performance can drift as a list ages or a lookalike source changes. Recording it lets you compare a customer profile's early performance against its later performance, or tell two versions of "the same" customer profile apart. Customer profile type and customer profile seed work together rather than separately. Customer profile type says what kind of customer profile it is (a lookalike, a retargeting pool, a customer list); customer profile seed says where it came from. Two 3% lookalikes built from different seed lists can perform very differently, so recording both is what lets you attribute a result to the customer profile itself rather than just to "lookalikes" in general.
</Accordion>

<Accordion title="Concept review: additional targeting and segment categories">
  Geo, language, and device only earn a place in the name if you actually target or create by them. An account running only broad, English-language, all-device campaigns doesn't need any of the three; one that runs separate creative for mobile and desktop, or localizes into French, does. List and segment type is the ad-set-level equivalent for email and SMS. A newsletter send and a cart-abandonment trigger aren't different customer profiles in the ad-platform sense, but they're different sends with different timing and intent, and naming them the same way you'd name a Meta ad set keeps them comparable in reporting.

  If you add one of these to your names, decide where it sits in the order and keep it there consistently.
</Accordion>

## Assemble the name

Spaces, `%`, and other special characters can break platform parsing or reporting exports. Use the short form you logged above for anything that isn't a plain alphanumeric string.

- Long form: `20260601-3-Percent-LLA-HubSpot-Leads`
- Short form (name jammed): `20260601-LLA3-HSL`

## Testing identifiers

<Tip>
  Optional: append only to sets / groups / variants that are part of an active test.
</Tip>

Testing identifiers mark a hypothesis you're currently running, for example an A/B variant test. Because a test only applies to some sets / groups / variants for a limited time, testing identifiers aren't part of the standard identifier set: append one to the end of the sets / groups / variant name, after the standard identifiers, instead of building it into a table above.

- With a testing identifier appended: `20260601-3-Percent-LLA-HubSpot-Leads-TEST01`

This document only covers where a testing identifier sits in the name. It doesn't cover how an [experiment ID](/glossary#experiment-id) gets created, how long it stays on an ad, or where the result gets logged once the test ends. Visit the [Experiment ID SOP](/experiment-ids) for that.
